# Архитектура мода умных поездов

## Цель архитектуры

Архитектура должна позволить сделать мод итерационно:

1. Сначала минимальный рабочий диспетчер поездов.
2. Потом умные хранилища и конвейерные модули.
3. Потом расширение планировщика, GUI и баланс.

Главный принцип: игровая механика живет в отдельных сервисах, а Factorio API
используется только на краях системы. Это нужно, чтобы диспетчер, планировщик и
резервации можно было тестировать и менять без переписывания всего мода.

## Lua и внешние библиотеки

Runtime-часть мода должна быть написана на Lua. Factorio запускает моды в своем
детерминированном Lua-окружении, поэтому подключать DLL/native-библиотеки из
мода нельзя считать рабочей архитектурой.

Что это значит для проекта:

1. `control.lua` и все, что выполняется внутри игры, пишется на Lua.
2. Планировщик, поиск, резервации и комбинаторика тоже должны иметь Lua-версию,
   если они используются во время игры.
3. Нельзя рассчитывать на `dll`, `so`, `package.loadlib`, `io`, `os`,
   `loadfile`, внешние процессы или чтение произвольных файлов во время runtime.
4. Нативная библиотека сломала бы детерминизм, мультиплеер, переносимость и
   совместимость с мод-порталом.

При этом алгоритмическое ядро стоит писать как отдельную библиотеку без Factorio
API:

```text
scripts/
  core/
    request_builder.lua
    offer_builder.lua
    candidate_filter.lua
    route_ranker.lua
    load_planner.lua
    planner_core.lua
    cost_model.lua
    combinator.lua
    dispatcher_core.lua
    table_utils.lua
```

Правила для `core/`:

1. Не использовать `game`, `script`, `storage`, `LuaEntity`, `LuaTrain`.
2. На вход принимать обычные таблицы Lua.
3. На выход возвращать обычные таблицы Lua: план, стоимость, причину отказа.
4. Не иметь побочных эффектов.
5. Не читать и не писать Factorio state напрямую.

Текущий pipeline ядра:

```text
station table
  -> request_builder: station_id + requested res[]
  -> offer_builder: station_id + available res[]
  -> candidate_filter: источники с хотя бы одним нужным ресурсом
  -> route_ranker: стоимость пути и приоритеты станций
  -> load_planner: упаковка ресурсов по поездам и станциям погрузки
  -> dispatcher_core: общий результат для запроса
```

Комбинаторика и оптимизация не должны обращаться к сырым станциям напрямую.
Они получают изолированные DTO, поэтому позже можно заменить алгоритм без
переписывания регистрации станций и чтения сигналов.

Такое ядро можно тестировать вне Factorio обычным Lua-runner'ом и при желании
дублировать экспериментальные версии на другом языке для сравнения результатов.
Но версия, которая реально работает в моде, должна быть Lua.

Допустимые внешние инструменты:

1. Генератор тест-кейсов.
2. Бенчмарк планировщика.
3. Скрипт сравнения Lua-эвристики с ILP/CP-SAT решением.
4. Генератор статических таблиц/весов.
5. Документация и визуализация планов.

Недопустимо для runtime:

1. Вызывать DLL из мода.
2. Вызывать внешний `.exe`.
3. Делать IPC/HTTP к локальному сервису.
4. Читать файлы с диска для принятия игровых решений.
5. Делать план, который в мультиплеере может дать разный результат на разных
   машинах.

## Слои системы

### Prototype layer

Отвечает только за объекты игры:

1. Умная станция.
2. Умный поезд.
3. Умное хранилище.
4. Временное хранилище.
5. Модуль входа с конвейера.
6. Модуль выхода на конвейер.
7. Технологии, рецепты, предметы, сигналы.

Здесь не должно быть логики диспетчера. Только описание сущностей, предметов,
рецептов и базовых параметров.

### Runtime layer

Отвечает за состояние мира во время игры:

1. Регистрация построенных станций.
2. Регистрация умных поездов.
3. Регистрация хранилищ.
4. Чтение сигналов.
5. Создание запросов.
6. Планирование доставок.
7. Назначение маршрутов.
8. Контроль выполнения.
9. Очистка после удаления сущностей.

### Integration layer

Тонкая прослойка над Factorio API:

1. Получить сигналы станции.
2. Проверить содержимое поезда.
3. Поставить расписание поезду.
4. Проверить, прибыл ли поезд.
5. Прочитать лимит станции.
6. Вывести сигнал или ошибку игроку.

Остальные модули не должны напрямую знать детали Factorio API, если это можно
избежать.

## Основные модели данных

### Station

```lua
Station = {
  id = 1,
  unit_number = 123,
  entity = LuaEntity,
  surface_index = 1,
  force_index = 1,

  mode = "load" | "unload",
  priority = 0,
  enabled = true,

  request_mode = "normal" | "prefer-buffer" | "only-buffer",
  buffer_mode = false,
  send_only_to_buffer = false,

  train_limit = 1,
  current_train_count = 0,

  linked_storage_ids = {},

  manual_requests = {},
  active_request_ids = {},
  active_delivery_ids = {},

  last_signals = {},
  last_error = nil
}
```

Станция должна быть либо `load`, либо `unload`. Одновременно погрузка и разгрузка
запрещены.

### SmartTrain

```lua
SmartTrain = {
  id = 1,
  train = LuaTrain,
  surface_index = 1,
  force_index = 1,

  state = "idle" | "assigned" | "to-source" | "loading" | "to-target" |
          "unloading" | "returning" | "error",

  only_buffer_sources = false,
  base_schedule = nil,
  active_delivery_id = nil,

  last_known_station_id = nil,
  last_error = nil
}
```

Перед назначением заказа поезд должен быть пустым. Если поезд не пустой, он
остается вне пула доступных поездов.

### Request

```lua
Request = {
  id = 1,
  station_id = 10,
  surface_index = 1,
  force_index = 1,

  priority = 0,
  created_tick = 1000,
  state = "open" | "planned" | "partial" | "complete" | "error",

  items = {
    ["iron-plate"] = 4000,
    ["copper-plate"] = 2000
  },

  remaining_items = {},
  planned_items = {},
  delivered_items = {},

  condition = {},
  source_policy = "normal" | "prefer-buffer" | "only-buffer",

  delivery_ids = {},
  last_error = nil
}
```

Запрос закрывается только когда `remaining_items` пустой или все значения равны
нулю.

### Delivery

```lua
Delivery = {
  id = 1,
  request_id = 1,
  train_id = 20,

  state = "planned" | "assigned" | "to-source" | "loading" | "to-target" |
          "unloading" | "complete" | "cancelled" | "error",

  source_station_ids = { 2, 3 },
  target_station_id = 10,

  load_plan = {
    {
      station_id = 2,
      items = {
        ["copper-plate"] = 800,
        ["electronic-circuit"] = 200
      }
    },
    {
      station_id = 3,
      items = {
        ["iron-plate"] = 800
      }
    }
  },

  loaded_items = {},
  delivered_items = {},

  created_tick = 1000,
  assigned_tick = 1010,
  started_loading_tick = nil,
  last_error = nil
}
```

Одна доставка может иметь несколько станций погрузки, но планировщик должен
штрафовать это и предпочитать одну станцию, если она может выдать нужный набор.

### Reservation

```lua
Reservation = {
  id = 1,
  delivery_id = 100,
  request_id = 1,
  train_id = 20,

  source_items = {
    [2] = {
      ["copper-plate"] = 800,
      ["electronic-circuit"] = 200
    }
  },

  target_items = {
    ["copper-plate"] = 800,
    ["electronic-circuit"] = 200
  },

  state = "active" | "released"
}
```

Резервации нужны, чтобы один и тот же сигнал доступности источника не был
использован несколькими поездами одновременно.

## Основной цикл диспетчера

Диспетчер не должен пересчитывать весь мир каждый тик.

Предлагаемый цикл:

1. Каждые `N` тиков обновлять небольшую порцию станций.
2. Считывать сигналы и обновлять запросы.
3. Добавлять новые запросы в очередь.
4. Выбирать ограниченное число запросов для планирования.
5. Строить план доставок.
6. Назначать доступные пустые поезда.
7. Проверять выполнение активных доставок.
8. Освобождать резервации завершенных или отмененных доставок.

Начальные значения для MVP:

1. Обновление станций: раз в 60 тиков.
2. Планирование: максимум 5-10 запросов за проход.
3. Проверка поездов: раз в 30 тиков.
4. Максимум активных доставок на поверхность: ограничить настройкой.

Эти числа нужны только как стартовые. После тестов их надо балансировать.

## Событийная модель

### События построения

При создании умной станции:

1. Выдать уникальный `station_id`.
2. Зарегистрировать станцию в `storage`.
3. Привязать surface/force.
4. Создать дефолтные настройки.
5. Добавить станцию в индекс поверхности.

При создании умного поезда:

1. Зарегистрировать поезд.
2. Сохранить базовое расписание, если оно есть.
3. Проверить, пустой ли поезд.
4. Добавить поезд в пул доступных, если он подходит.

При создании умного хранилища:

1. Зарегистрировать хранилище.
2. Попытаться привязать к ближайшей умной станции.
3. Если привязка неоднозначна, оставить ошибку/состояние `unlinked`.

### События удаления

При удалении станции:

1. Отменить незапущенные запросы станции.
2. Активные доставки к этой станции перевести в ошибку или отмену.
3. Освободить резервации.
4. Удалить станцию из индексов.

При удалении поезда:

1. Отменить его активную доставку, если загрузка еще не началась.
2. Если поезд уже везет ресурс, перевести доставку в ошибку.
3. Освободить резервации.

При удалении хранилища:

1. Убрать связь со станцией.
2. Если поезд ждал загрузку через это хранилище, перевести доставку в ошибку.

## Планировщик доставок

Планировщик получает:

1. Запрос.
2. Список доступных пустых поездов.
3. Список подходящих станций-источников.
4. Доступные ресурсы источников с учетом резерваций.
5. Лимиты станций.

Планировщик возвращает:

1. Список доставок.
2. Список резерваций.
3. Оценку стоимости плана.
4. Причину отказа, если план невозможен.

### Стоимость плана

```text
cost =
  mixing_weight * extra_resource_types_per_train +
  train_weight * train_count +
  station_weight * load_station_count +
  empty_space_weight * empty_capacity
```

Стартовые веса:

1. `mixing_weight = 1000`
2. `train_weight = 500`
3. `station_weight = 100`
4. `empty_space_weight = 1`

Это не финальный баланс, а способ сделать решения планировщика предсказуемыми.

### MVP-алгоритм

Для первой версии не надо сразу писать точный оптимизатор.

Практичная эвристика:

1. Отсортировать ресурсы запроса по объему от большего к меньшему.
2. Отсортировать доступные поезда по вместимости от большего к меньшему.
3. Сначала попытаться закрыть крупные ресурсы цельными партиями.
4. Затем добивать свободное место меньшими ресурсами.
5. Для каждого кандидата считать стоимость.
6. Сравнить несколько лучших кандидатов, а не брать первый.
7. Не трогать уже отправленные поезда при перепланировании.

Если эвристика начнет плохо работать, ее можно заменить на более строгий
оптимизатор без изменения остальной архитектуры.

### Возможное точное решение

В будущем планировщик можно заменить на:

1. ILP/MILP для небольших запросов.
2. CP-SAT-подобную модель, если удастся реализовать или портировать.
3. Min-cost flow для части задачи с назначением ресурсов источникам и поездам.
4. Гибрид: жадная генерация кандидатов + локальный поиск.

Для Factorio Lua не стоит сразу тянуть тяжелую математику в runtime. Сначала
нужен понятный интерфейс планировщика, потом можно улучшать реализацию.

## Executor

`executor.lua` отвечает за перевод плана в действия поезда.

Задачи:

1. Сохранить базовое расписание поезда.
2. Поставить временное расписание доставки.
3. Отправить поезд к первой станции погрузки.
4. На каждой станции погрузки передать станции/хранилищу список предметов для
   загрузки.
5. Дождаться выполнения загрузки.
6. Отправить поезд к следующей станции погрузки или к целевой станции.
7. Дождаться разгрузки.
8. Закрыть доставку.
9. Вернуть поезд к базовому расписанию или ожиданию.

Важное правило: после начала загрузки доставка не отменяется из-за изменения
сигнала источника. Если ресурс фактически не подается, поезд ждет.

## Circuit integration

Умная станция должна иметь вход и выход логической сети.

Вход станции:

1. Красный провод источника объявляет доступные ресурсы.
2. Сигналы запроса задают требуемые ресурсы и количества.
3. Условие запроса задается как comparator-условие в GUI станции.

Выход станции:

1. Активный запрос.
2. Ожидаемые доставки.
3. Состояние станции.
4. Ошибка станции.

Для MVP можно вывести минимум:

1. Активный запрос.
2. Ожидаемую доставку.
3. Ошибку через служебный сигнал.

## GUI

GUI должен быть простым, иначе мы утонем до рабочего прототипа.

### Умная станция

Нужные элементы:

1. Режим станции: `погрузка` / `разгрузка`.
2. Приоритет.
3. Лимит поездов.
4. Режим источников: `обычные`, `предпочитать буфер`, `только буфер`.
5. Галочка `буферная станция`.
6. Ручные строки заказа: ресурс, количество, условие.
7. Отображение активных доставок.
8. Последняя ошибка.

### Умный поезд

Нужные элементы:

1. Включить/выключить участие в системе.
2. Брать только с буферных станций.
3. Текущее состояние.
4. Активная доставка.
5. Последняя ошибка.

### Умное хранилище

Нужные элементы:

1. Привязанная станция.
2. Содержимое.
3. Текущее задание на загрузку.
4. Ошибка связи со станцией.

## Привязка хранилища к станции

Для MVP предлагаю простой вариант:

1. Умное хранилище ищет ближайшую умную станцию в радиусе.
2. Если найдена ровно одна станция, хранилище привязывается к ней.
3. Если станций нет или их несколько, хранилище переходит в `unlinked`.
4. В GUI хранилища показывается ошибка.

Позже можно добавить ручную привязку проводом или инструментом.

## Производительность

Основные риски:

1. Частое чтение сигналов всех станций.
2. Полный перебор всех источников для каждого запроса.
3. Сложное планирование мульти-ресурсных заказов.
4. Постоянная проверка всех поездов.

Защита:

1. Индексы по поверхности и force.
2. Индексы источников по ресурсу.
3. Инкрементальное обновление станций.
4. Лимит запросов на планирование за проход.
5. Кэш доступности источников на короткое время.
6. Простая эвристика планирования в MVP.
7. Отложенная переоценка, если запрос невозможно выполнить.

## Индексы

Нужные индексы:

```lua
sources_by_item[surface_index][force_index][item_name] = { station_id, ... }
requests_by_priority[surface_index][force_index] = { request_id, ... }
trains_by_surface[surface_index][force_index] = { train_id, ... }
stations_by_surface[surface_index][force_index] = { station_id, ... }
storages_by_station[station_id] = { storage_id, ... }
```

Индексы должны перестраиваться при:

1. Создании/удалении станции.
2. Изменении режима станции.
3. Изменении сигнала доступности источника.
4. Изменении force/surface.
5. Миграции версии.

## Ошибки и диагностика

Ошибки не должны ломать весь диспетчер.

Типы ошибок:

1. `no-source`: нет источника.
2. `no-train`: нет пустого доступного поезда.
3. `source-lost`: источник потерял ресурс до загрузки.
4. `target-lost`: целевая станция удалена или отключена.
5. `train-lost`: поезд удален или стал невалидным.
6. `storage-unlinked`: хранилище не привязано к станции.
7. `loading-timeout`: поезд слишком долго ждет загрузку, если включим таймаут.

В MVP достаточно хранить последнюю ошибку на станции, поезде и доставке.

## Порядок реализации MVP

### Этап 1: Каркас мода

1. `info.json`.
2. `data.lua`.
3. Прототип умной станции.
4. Прототип умного поезда.
5. Простейшие рецепты и технология.
6. Регистрация построенных/удаленных сущностей.

## Что я бы не делал в первой версии

1. Жидкости.
2. Межпланетные доставки.
3. Полноценный математический оптимизатор.
4. Сложный GUI диспетчера на весь мир.
5. Поддержку всех модовых вагонов и локомотивов.
6. Автоматический ремонт плохой рельсовой сети игрока.

## Главные технические риски

1. Насколько удобно и надежно управлять расписанием поезда без конфликтов с
   ванильными interrupt-механиками.
2. Как лучше сделать умную станцию: копировать обычную станцию или оборачивать
   обычную станцию скрытой логикой.
3. Как именно реализовать вход/выход логической сети у станции.
4. Как физически грузить поезд через умное хранилище без слишком тяжелой логики.
5. Как не убить производительность планировщиком на больших базах.

## Моя рекомендация

Начинать с минимального вертикального среза:

1. Одна поверхность.
2. Один force.
3. Умная станция погрузки.
4. Умная станция разгрузки.
5. Один умный поезд.
6. Один ресурс.
7. Ручной запрос без сложного GUI.
8. Назначение временного маршрута.
9. Закрытие доставки.

После этого уже добавлять сигналы, несколько ресурсов, планировщик и хранилища.
Так мы быстро проверим самый рискованный кусок: можно ли стабильно управлять
поездами так, как требует механика.
